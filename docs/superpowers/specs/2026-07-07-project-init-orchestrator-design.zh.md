# 项目初始化编排器设计

## 目标

创建一个可复用的双平台工作流包，支持 Codex 和 Claude Code，让每个新项目在正式实现之前都必须先完成规范化的项目初始化。这个工作流应当让主 Agent 先建立项目目标、收集需求、编写项目控制文档、定义子 Agent 边界、创建工作日志，然后进入编排式循环，持续推进项目，直到项目完成或确实遇到阻塞。

这个工作流会刻意保持强约束。它宁可在前期多做需求澄清和边界记录，也不要让 Agent 过早开始写代码。它主要要解决的问题是：Agent 在项目范围、验收标准和职责边界尚未稳定之前就直接进入实现，导致后续返工、越界和混乱。

## 目标用户

主要用户是希望在 Codex、Claude Code 或类似 Agent 编程环境中，以可重复方式启动新项目的操作者。用户预期可以从 GitHub 下载或克隆这个仓库，安装工作流，然后在新的项目对话中直接触发它。

次要用户是另一个 Agent。Skill 文件和命令文件必须写得足够明确，让一个全新的 Agent 不依赖当前对话上下文，也能正确执行这个流程。

## 交付物

仓库将包含：

1. Codex Skill：`codex/project-init-orchestrator/SKILL.md`。
2. Codex UI 元数据：`codex/project-init-orchestrator/agents/openai.yaml`。
3. Claude Code 命令：`claude-code/commands/project-init-orchestrator.md`。
4. 共享模板目录：`templates/`。
5. 安装脚本：`install.sh`。
6. 面向 GitHub 用户的 `README.md`。
7. 必要的轻量验证脚本或检查。

Codex Skill 是 Codex 里的主要可执行工作流入口。Claude Code 命令是 Claude Code 里的原生入口。共享模板用于保证两个平台生成的项目文档结构一致。

## 工作流契约

当工作流在一个项目中被触发时，主 Agent 必须按以下顺序执行：

1. 创建或确认一个持久化项目目标。如果平台有原生 goal 模式，就使用原生 goal 模式；如果没有，就通过项目工作日志记录持久化目标和循环状态来模拟 goal 模式。
2. 启用循环式执行模型：澄清、设计、文档化、计划、执行、验证、复盘，并持续循环，直到项目完成或出现已证实的阻塞。
3. 在做任何假设之前，先检查当前项目上下文。
4. 在可用时调用 `superpowers:brainstorming`，并遵守它的 gate：设计未获批准之前不得进入实现。
5. 向用户提出聚焦问题，覆盖项目目的、目标用户、范围、约束、成功标准、风险、集成点和偏好的执行方式。
6. 生成或更新项目级 `AGENTS.md`。
7. 在 `docs/specs/` 下生成项目 spec。
8. 生成任务计划和工作日志文件。
9. 在派发任何子 Agent 工作之前，先定义子 Agent 角色和 brief。
10. 确保子 Agent 只能依据 `AGENTS.md`、项目 spec 和自己的 brief 工作。
11. 要求所有 Agent 在工作日志中记录进度、决策、阻塞和交接事项。
12. 主 Agent 负责整合子 Agent 结果、解决冲突、更新计划，并继续推进循环。
13. 在宣布完成之前，必须基于 goal、spec、任务列表、测试和生成产物进行逐项完成审计。

工作流不能假设所有平台都有完全相同的运行能力。如果某个平台没有原生子 Agent、goal 模式或 skill 调用能力，命令必须退化为使用明确的角色 brief 和持久化日志，但仍然保持同样的流程纪律。

## 生成的项目产物

每个被初始化的项目默认应获得以下文件，除非用户或现有仓库规范要求使用等价路径：

- `AGENTS.md`：所有 Agent 都必须遵守的项目宪章。
- `docs/specs/<date>-project-spec.md`：需求、架构、范围、风险、验收标准和测试策略。
- `docs/worklogs/main-worklog.md`：主 Agent 的循环状态、决策、进度和完成审计。
- `docs/worklogs/subagents/<role>-worklog.md`：每个子 Agent 角色各自的工作日志。
- `docs/agents/subagent-briefs/<role>.md`：每个子 Agent 的受限工作包。
- `docs/tasks/project-tasks.md`：任务拆解、负责人、状态、依赖关系和验证证据。

## AGENTS.md 要求

生成的 `AGENTS.md` 必须定义：

- 项目目标和非目标。
- 已发现的技术栈和本地命令。
- 目录职责和允许编辑区域。
- 编码、测试、文档和评审标准。
- 主 Agent 职责。
- 子 Agent 角色边界。
- 工作日志要求。
- 决策和升级规则。
- 禁止行为，包括擅自扩展范围、未记录的架构变更，以及编辑授权范围之外的文件。

## Spec 要求

项目 spec 必须定义：

- 背景和问题陈述。
- 目标用户和核心使用场景。
- 范围内和范围外事项。
- 功能需求。
- 非功能需求。
- 架构和模块边界。
- 数据流和外部依赖。
- 错误处理和边界情况。
- 测试与验证策略。
- 验收标准。
- 风险、假设和开放问题。

在进入实现之前，spec 必须经过自审，检查占位符、矛盾、模糊需求和失控范围。

## 子 Agent 协作模型

主 Agent 拥有编排权。它决定何时创建或模拟子 Agent，编写每个 brief，检查子 Agent 输出，并把结果合并回项目计划。

每个子 Agent 必须收到：

- 角色名称。
- 精确目标。
- 允许编辑的文件和目录。
- 禁止触碰的文件和禁止自行决定的事项。
- 必需输入。
- 预期输出。
- 必需验证。
- 工作日志位置。
- 交接要求。

子 Agent 可以通过共享工作日志和明确交接进行协作。子 Agent 不能静默修改其他角色负责的区域，也不能扩大项目范围。

## 循环模型

循环必须持久化并记录在文档中：

1. 读取当前 goal、spec、任务和工作日志。
2. 识别下一步最高价值工作。
3. 派发或执行受限任务。
4. 记录决策和证据。
5. 验证结果。
6. 更新任务和日志。
7. 持续循环，直到完成审计通过或阻塞被证实。

在项目细节已经收集完成之后，这个循环不应要求用户手动推动常规下一步。只有当缺失信息会影响范围、风险、产品方向、凭据、外部访问或验收标准时，才需要用户再次输入。

## 安装设计

仓库同时支持手动安装和脚本安装。

`install.sh` 脚本将：

- 检测仓库根目录。
- 将 Codex Skill 安装到 `${CODEX_HOME:-$HOME/.codex}/skills/project-init-orchestrator`。
- 将 Claude Code 命令安装到 `$HOME/.claude/commands/project-init-orchestrator.md`。
- 在需要时创建目标目录。
- 保留仓库中的源码文件。
- 打印安装路径和触发示例。

用户也可以通过复制同样的目录和文件进行手动安装。

## README 范围

仓库的 `README.md` 将说明：

- 这个工作流做什么。
- 它会安装什么。
- 如何安装。
- 如何在 Codex 和 Claude Code 中触发。
- 它会在目标项目中创建哪些文件。
- goal 模式、loop 模式、子 Agent 边界和工作日志如何工作。
- 平台限制和降级方案。

## 验证

验证应证明：

- Codex Skill 有有效的 frontmatter，并且能通过 bundled skill validator。
- 必需文件存在于预期仓库路径。
- 安装器可以在 dry-run 或临时 HOME 模式下运行，不会破坏用户真实安装。
- 模板包含 goal 模式、loop 模式、spec、子 Agent brief 和工作日志所需的章节。

## 当前决策

第一版将使用英文文件内容，以便更适合 GitHub 上的广泛复用；但工作流本身仍然适合中文项目对话。未来版本可以增加双语模板。

第一版不实现单独的运行时二进制程序。行为将存在于平台原生指令文件、共享模板和安装脚本中。

## 验收标准

当以下条件全部满足时，目标才算完成：

1. 仓库包含 Codex Skill、Claude Code 命令、模板、安装器和 README。
2. Codex Skill 明确要求主 Agent 使用 goal 模式、循环执行、`superpowers:brainstorming`、`AGENTS.md`、spec、受限子 Agent、工作日志和完成审计。
3. Claude Code 命令以 Claude Code 原生形式提供同一套工作流。
4. 工作流会创建的所有项目产物都有对应模板。
5. 安装方式有文档说明，并且安装脚本可执行。
6. 验证命令通过。
7. 完成审计确认当前仓库里的证据覆盖本设计中的每一项要求。
